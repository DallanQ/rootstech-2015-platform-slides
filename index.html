Personal Family-Trees

Owner decides who to let in

  • Follow google docs permission structure
  • read, comment, edit
  • specific users or public (minus living)

Suppose you want to create a family tree tool

  • Not a full-blown application
  • Visualization (puzilla), photo editing, research guidance (find-a-record)

Options

  • Build on top of FamilySearch - limited user base
  • Build on top of MyHeritage - read-only
  • Build on top of Ancestry Trees - no API
  • Build your own tree manager
    • - lots of foundation code to write
    • - grow user base ( marketing $ )
  • Many (most) people don't want to use a shared tree as their primary database
  • foundation code before you get to your idea
  • not like pinterest; users must trust you with sensitive information in order to get started

We need a Platform

For personal family trees, like FamilySearch is a platform for a shared family tree

Platform Benefits

  • Foundation code written
  • GEDCOM import
  • GEDCOM export
  • REST API
  • OAuth support
  • you don't have to write the boilerplate code
  • saves months of work
  • free hosting
  • don't worry about managing database
  • OAuth support means users can have multiple apps hitting the same database
  • single database, multiple applications
  • you don't have to spend marketing $ to grow your own user-base
  • Like writing a wordpress plugin vs creating an entire blog system from scratch

More Benefits (later)

  • Open-source web, desktop, and mobile clients
  • Plug your tools into these clients
  • Clients written using web components
  • Embed components into your own app
  • All javascript: ionic framework, node-webkit
  • significant effort
  • allows sharing code
  • start with open-source clients and write plugins for your ideas
  • or use components from the open-source clients in your own applications

Evidence-Based Data Model

  • Data model includes events as first-class objects
  • Tom Wetmore (DeadEnds), Tony Proctor (STEMMA), Louis Kessler, Tom Wetmore, Randy Wilson
  • current: conclusions, then sources (typically two clicks to see record data)
  • this: sources (records), then conclusions

Evidence-Based Advantages

  • More fun
  • Better model for collaboration
  • Increase chances to get new people involved
  • let's face it: conclusions themselves are boring, how we got those conclusions - records - are interesting
  • View, comment on, and share records, not just conclusions
  • Einstein: The definition of insanity is doing the same thing over and over and expecting different results.
  • a lot of effort trying to get more people interested in the genealogy game; maybe it's time to change the way the game is played

Why Adopt the Platform?

Advantages

  • Back-end code written
  • Less custom code to write
  • Don't need to spend money growing your user base
  • Free, hosted service
  • Simple REST API
  • Evidence-based approach

Disadvantages

  • Depend upon external entity
  • Loss of control over back-end storage
  • I'm hoping to greatly increase the family history application ecosystem.
  • Make it much, much easier to write applications
  • Like writing a wordpress plugin vs creating an entire blog system from scratch
  • Disadvantage: like using github instead of hosting your own version control system
  • The back-end code is not open-source

REST-ish API

  • GET, POST, PUT, DELETE verbs
  • No unique content types - everything is application/json
    • If you request a person, you get back a person
    • If you request an event, you get back an event
  • API version is part of the request URI
  • HATEOS?
  • matches what you typically see in Web APIs
  • content headers don't work well in browsers - redirects
  • Currently I haven't implemented HATEOS
  • worry about links becoming stale
  • could be talked into it though
